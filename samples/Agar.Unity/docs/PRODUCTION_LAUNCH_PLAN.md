# Agar.Unity 生产上线规划

本文面向 `samples/Agar.Unity` 从“可演示样例”推进到“小规模生产上线”的评审。现有阶段计划见 `DEVELOPMENT_PLAN.md`；本文只回答上线前还缺哪些功能、还需要做哪些工程和验证工作。

## 上线目标

首个生产版本应定位为轻量小球吞噬对战：

- 单机模式可离线完整游玩。
- 联机模式支持账号/游客登录、匹配、实时对局、结算、胜利积分和周榜。
- 服务端权威推进房间模拟，客户端只发送输入并渲染快照。
- 网关、Silo、PostgreSQL 和 Redis 可以通过 Docker 在非本机环境部署、监控、回滚和排障。
- 玩家界面不暴露调试信息，基础美术、中文文案和核心流程达到可交付标准。

不把分裂、吐质量、病毒、组队、复杂技能树、任务、商店和长期历史榜单纳入首发范围。

## 当前状态判断

当前样例已经具备核心玩法和联机骨架：共享模拟内核、WebSocket 控制面、KCP 实时面、Orleans 匹配与房间分配、可靠匹配推送、PostgreSQL grain 状态保存、胜利积分和排行榜都已接入。生产上线前，胜利积分排行榜索引计划迁移到 Redis sorted set。

距离生产上线的主要差距不在“有没有基本玩法”，而在以下几类：

- 跨网关实时路由尚未完成，活跃房间模拟和世界状态扇出仍局限在单个网关进程。
- 断线、登出、取消匹配、离房和空房间释放的清理语义还需要集中验证和补测试。
- Unity 客户端主流程仍有较高维护风险，`DotArenaGame` 和 `DotArenaSceneUiPresenter` 还没有完全收敛到清晰边界。
- 生产基础设施还停留在只启动 PostgreSQL/Redis 的本地 compose 和开发配置，缺少 Edge/Silo 镜像、生产 compose、真实环境配置、密钥、日志、监控、告警和发布流水线。
- 自动化测试覆盖了模拟、匹配策略、排行榜和部分会话清理，但缺少房间运行时、网络会话生命周期、客户端流程和端到端回归。

## 必须补齐的玩家功能

首发前必须补齐或明确验收的玩家功能如下。

| 模块 | 当前基础 | 上线前缺口 | 首发处理 |
| --- | --- | --- | --- |
| 登录与会话 | 已有登录、游客登录、重连参数和 token | 缺少异常状态的玩家级提示、重复登录策略、token 生命周期和状态丢失口径 | 必须补齐 |
| 匹配 | 已有 Orleans 匹配队列和可靠匹配推送 | 取消匹配、断线重连、匹配超时、重复点击的最终体验需要回归 | 必须补齐 |
| 实时对局 | 已有输入提交、服务端模拟、世界快照和结算 | 跨网关输入转发和世界状态回送未实现；网络抖动下的表现和降级策略不足 | 必须补齐 |
| 断线恢复 | 已有可靠推送 ack 和重连状态丢失码 | 需要完整覆盖控制连接断开、实时连接断开、房间结束后重连、outbox 过期后的体验 | 必须补齐 |
| 排行榜 | 已接服务端真实排行榜和周重置 | 排行榜索引需要迁移为 Redis sorted set；榜单时区、归档和 Redis 不可用行为需要产品化 | 必须补齐 |
| UI 与美术 | 已有中文 UI、基础素材和低遮挡 HUD 排名 | 需要完成 1200x600、960x540 和目标发布分辨率的人工视觉回归 | 必须补齐 |
| 新手引导 | 只有基础操作说明 | 玩家首次进入缺少最小引导、连接失败说明和返回路径 | 建议首发补最小版 |
| 设置 | 已有基础设置入口 | 缺少音量、画质、网络区域、账号退出等稳定设置项 | 可做最小版 |

首发功能边界建议保持克制：不要在上线前重新打开任务、商店、记录、装备养成或复杂皮肤系统。当前更需要把登录、匹配、断线、对局、结算和排行榜这一条主链路做稳。

## 必须完成的服务端工作

### 分布式实时路由

当前房间运行时、RPC 回调对象和世界状态扇出仍在单个网关进程内。生产环境只要有多网关，就会遇到控制连接在网关 A、房间运行时在网关 B 的情况。

必须完成：

- 选定网关间事件机制：优先评估 Orleans stream 或 Redis 发布订阅，明确输入转发、世界状态扇出、断线事件和背压所有权。
- 确保房间模拟同一时刻只有一个权威 runtime owner。
- 输入消息跨网关投递必须有顺序和过期策略，不能无限排队。
- 世界状态广播失败时要有可观测日志，不能悄悄丢玩家。
- 不能把实时 RPC callback 对象序列化进 Redis 或 Orleans grain 状态。

验收门槛：

- 两个 Edge 实例运行时，玩家控制连接、实时连接和房间 runtime 可以分布在不同网关。
- 玩家输入能到达 runtime owner，世界状态能回到正确客户端。
- 关闭其中一个 Edge 后，受影响玩家得到明确失败/重连体验，其他房间不受影响。

### 清理与恢复语义

必须集中梳理 `PlayerService`、`SessionRegistration`、`SessionDirectory`、`DisconnectedSessionCleanupHostedService`、`RoomRuntimeHost` 和 `RoomRuntime`。

必须完成：

- 登录、登出、重复登录、取消匹配、匹配超时、实时断线、控制断线、离房和对局结束都有幂等清理路径。
- 控制连接和实时连接可以独立解绑。
- 匹配票据、实时注册、房间玩家、可靠推送 pending 记录和本地 callback 状态不会永久残留。
- 空房间可以释放 runtime 资源。
- outbox 过期或 server state lost 时，客户端必须被要求开启新会话。

### 持久化与数据模型

必须完成：

- 把开发连接串、端口、节点 id 和 advertised IP 从本地配置迁移到环境变量或部署配置。
- 为排行榜时区、周期重置、归档保留周期、房间容量、匹配超时、可靠推送保留时间提供配置项。
- 明确 PostgreSQL schema 迁移流程，不能依赖手动复制 init SQL 作为生产升级方式。
- 将当前周期胜利积分排行榜索引迁移到 Redis sorted set，PostgreSQL/Orleans grain 继续保存用户资料和持久化状态。
- 明确可靠业务推送使用内存短窗口，写清楚进程重启或 outbox 丢失后的 state lost / new session 玩家体验。
- 建立用户目录或补充排行榜重置策略。当前 leaderboard 只能重置索引内玩家，长期会和“所有玩家积分归零”的产品语义冲突。

### 安全与滥用防护

必须完成：

- 密码不能以明文或可逆形式持久化；游客账号也要有可撤销的会话凭据。
- RPC 请求要验证 playerId/token/roomId/matchId 的绑定关系，防止提交他人输入或查询越权数据。
- 登录、匹配、输入提交、排行榜查询需要基础限流。
- 服务端不要信任客户端 tick、位置、质量或排名。
- KCP 和 WebSocket 的公网暴露端口、TLS/反向代理、CORS/来源限制要有部署说明。

## 必须完成的客户端工作

### 主流程稳定化

必须继续阶段 6 的拆分，目标不是重构好看，而是降低上线前改 bug 的风险。

必须完成：

- `DotArenaGame` 收敛为场景组合根，不再直接拥有大量 UI、网络、模拟和表现字段。
- 联机登录、匹配、取消匹配、实时绑定、断线恢复和可靠 push ack 由单一流程对象编排。
- `DotArenaSceneUiPresenter` 拆出入口、登录、匹配、大厅、HUD、局内排名、结算等面板 presenter，避免一个类管理所有控件。
- 所有按钮在请求中、失败、超时和返回时有明确可点击状态，避免重复请求。
- UI 只展示玩家可理解状态，不展示 endpoint、tick、内部枚举或调试文本。

### 网络体验

必须补齐：

- 连接中、重连中、匹配中、取消中、对局已结束、服务器维护、状态丢失等玩家文案。
- 控制连接断开和实时连接断开的不同恢复路径。
- 可靠匹配推送重复到达时的幂等处理回归。
- 客户端本地保存 latest reliable sequence 的生命周期，确保新会话不会误 ack 旧会话序列。
- 对局中快照中断后的表现：短暂等待、重连提示、返回大厅或重新登录。

### 视觉与发布包

必须完成：

- 对入口、登录、匹配、大厅、HUD、局内排名、结算和排行榜做目标分辨率视觉回归。
- 检查所有中文文案不溢出，按钮和输入框在低分辨率下不遮挡。
- 确认素材许可、字体许可和第三方资源归属。
- 固定 Unity 版本、包版本和构建目标。
- 生成可复现的 Windows 或目标平台构建包，并记录 build 命令或 CI 步骤。

## 质量保障计划

自动化测试必须扩展到以下范围：

- `RoomRuntime`：玩家加入/离开、输入提交、世界快照、结算、空房间清理。
- `SessionDirectory` / `SessionRegistration`：控制注册、实时注册、重复解绑、房间查询、过期清理。
- `PlayerService`：登录、游客登录、重复登录、匹配、取消、实时绑定、ack state lost。
- `MatchmakingGrain` / `RoomGrain`：票据生命周期、房间分配、运行时网关端点、过期处理。
- Redis 排行榜 store / `LeaderboardGrain`：sorted set 写入、top N 查询、AI 过滤、周重置、归档、时区配置、Redis 不可用降级。
- 客户端流程：至少对 `DotArenaNetworkSession` 和流程状态机做纯 C# 测试。

手动回归必须形成固定清单：

- 单机完整一局：开始、移动、吃食物、吞噬、死亡、复活、结算、再来一局、返回入口。
- 联机完整一局：登录、匹配、实时绑定、移动、世界同步、结算、积分发放、排行榜刷新。
- 断线场景：匹配中断线、对局中控制连接断开、实时连接断开、服务端重启、重复登录顶号。
- 多网关场景：两个 Edge、一个 Silo，玩家分布在不同 Edge 完成一局。
- 视觉场景：1200x600、960x540 和目标发布分辨率。

上线候选版本的硬门槛：

- `dotnet build Shared/Shared.csproj -f net10.0` 通过。
- `dotnet build Server/Silo/Silo.csproj` 通过。
- `dotnet build Server/Edge/Edge.csproj` 通过。
- `dotnet test tests/BusinessLogic.Tests/BusinessLogic.Tests.csproj` 通过。
- Unity 脚本编译无错误。
- 单机和联机主链路人工回归通过。
- 多网关冒烟通过。

## 运维与发布工作

上线前必须补齐以下非功能工作：

- Docker 部署拓扑：Edge、Silo、PostgreSQL、Redis、反向代理或负载均衡、开放端口、Docker network、volume、健康检查和重启策略。
- 镜像产物：为 Edge 和 Silo 提供独立 Dockerfile 或统一多目标 Dockerfile，固定 .NET 运行时版本，使用非 root 用户，镜像 tag 包含版本号和 git commit。
- Compose 文件：保留本地 `docker-compose.yml` 作为开发基础设施；新增生产 compose 或 compose override，纳入 Edge、Silo、PostgreSQL、Redis、反向代理、env 文件和持久化 volume。
- 配置管理：环境变量、密钥、连接串、节点 id、公网 host、KCP 端口、榜单时区和日志级别。
- 日志：登录、匹配、房间创建、实时绑定、输入投递失败、世界状态广播失败、结算、积分发放、排行榜重置。
- 指标：在线人数、匹配队列长度、房间数、输入延迟、快照广播频率、断线率、RPC 错误率、数据库错误率。
- 告警：Edge 不健康、Silo 不健康、PostgreSQL/Redis 不可用、匹配队列积压、房间 runtime 异常退出、错误率升高。
- 数据备份：PostgreSQL 备份、恢复演练、排行榜归档策略。
- 发布流水线：服务端构建、镜像或可执行包产物、Unity 客户端构建、版本号、变更记录和回滚步骤。
- 灰度策略：先内部房间，再小规模外部测试，最后扩大流量。

| 风险 | 影响 | 处理 |
| --- | --- | --- |

## 上线工作概要

我们现在已经有一个可以演示的多人小球对战版本，核心玩法、登录、匹配、对局、结算和排行榜都有基础。下一阶段的目标不是继续堆新玩法，而是把这条主流程做成可以小规模上线、可以部署、可以监控、出问题能快速恢复的版本。

### 近期重点

1. 稳定玩家主流程：重点打磨登录、匹配、进入对局、断线重连、结算和排行榜，保证玩家不会卡在中间状态，也不会因为重复点击或网络波动导致流程混乱。
2. 补齐上线基础设施：把服务端做成可部署的 Docker 镜像，准备正式环境配置、数据库、缓存、日志、监控、告警、备份和回滚方案。
3. 支持小规模扩容：解决多台网关服务器之间的实时消息转发问题，确保玩家分布在不同服务器上时仍能正常匹配、移动、同步和结算。
4. 提升客户端可维护性：把 Unity 客户端中集中在大类里的登录、匹配、界面和网络逻辑拆清楚，降低上线前修 bug 时引入新问题的风险。
5. 做完整验收：补自动化测试和人工回归清单，覆盖单机、联机、断线、服务重启、多服务器、不同分辨率和发布包构建。

### 阶段安排

| 阶段 | 目标 | 交付结果 |
| --- | --- | --- |
| 第一阶段 | 冻结首发范围 | 明确首发只做登录、匹配、对局、结算、积分和排行榜，不再临时增加商店、任务、复杂技能等新内容 |
| 第二阶段 | 稳定单服务器版本 | 单台网关和单台业务服务器可以连续完成联机对局，断线、取消匹配、结算和排行榜流程稳定 |
| 第三阶段 | 准备生产部署 | 服务端镜像、正式配置、数据库、缓存、日志、监控、告警、备份、发布和回滚流程就绪 |
| 第四阶段 | 支持多服务器 | 两台以上网关服务器下，玩家仍能完成匹配、对局同步、断线恢复和结算 |
| 第五阶段 | 上线候选 | 自动化测试、人工回归、视觉检查、素材许可、发布演练全部通过，可以进入小规模外部测试 |

### 首发不做什么

首发建议克制范围，暂不做任务、商店、历史战绩、观战、回放、复杂皮肤、组队、分裂、吐质量和病毒等扩展玩法。这些内容会增加开发和测试成本，容易影响主流程稳定性，适合放到首发后的版本迭代。

### 主要风险

当前最大风险有四个：多服务器实时同步还未完成；断线、登出、取消匹配等异常流程需要继续验证；正式部署、监控和告警还不完整；账号凭据和请求校验需要达到外部上线的安全要求。只要这些问题没有关闭，就不建议直接对外生产上线。

### 汇报结论

这个项目已经过了“能不能做出来”的阶段，接下来要解决的是“能不能稳定上线”。后续工作应优先保证主链路稳定、服务端可部署可观测、多服务器可运行、客户端问题可快速修复。建议先完成小规模上线版本，再根据真实玩家反馈追加新玩法和商业化内容。
