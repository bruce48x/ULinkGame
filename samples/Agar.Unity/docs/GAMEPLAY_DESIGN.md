# 玩法与架构设计

这份文档记录样例游戏的长期设计状态。玩法规则、客户端服务端边界、联机流程、基础设施判断和暂不实现的内容都放在这里；阶段性任务安排放在 `DEVELOPMENT_PLAN.md`。

## 设计目标

样例要验证 ULinkGame 在一个轻量多人对战项目中的完整接入方式：

- 单机模式可以完全离线运行。
- 联机模式由服务端权威推进对局。
- 客户端只发送输入，服务端广播世界状态。
- 玩法规则只实现一份，并被单机和联机复用。
- 网络、界面、存档和部署细节不能污染玩法规则内核。

当前玩法参考轻量小球吞噬对战。刻意不做分裂、吐质量、病毒、尖刺、组队和复杂技能树，避免样例被玩法复杂度拖偏。

## 核心规则

玩家控制一个小球在方形场地中移动。

- `W/A/S/D` 控制移动方向。
- 移动是连续的。
- 体型越大，移动越慢。
- 协议里仍保留 `dash` 字段，但当前玩法不使用冲刺。

成长来源：

- 吃地图上的中立食物。
- 吞掉体型足够小的其他玩家。

每个玩家记录：

- `Mass`：权威成长值。
- `Radius`：由质量推导出的表现和碰撞半径。
- `MoveSpeed`：由质量推导出的移动速度。
- `Score`：用于界面展示和结算排序的本局贡献。

吞噬条件：

- 双方都存活。
- 距离进入吞噬范围。
- 吞噬者比目标大到配置要求的比例。

吞噬结果：

- 目标死亡并进入复活倒计时。
- 吞噬者获得质量。
- 吞噬者获得积分。

食物规则：

- 场地维持目标数量的食物。
- 食物被吃掉后会补回目标数量附近。
- 单机和联机使用同一套刷新规则。

复活规则：

- 玩家被吞后清空临时移动状态。
- 倒计时结束后重新出生。
- 复活时恢复初始质量和半径。
- 本局积分保留。

单局结束：

- 对局按倒计时结束。
- 排名优先按质量降序，其次按积分降序，最后按玩家标识升序作为稳定兜底。
- 结算后返回主界面。

## 胜利积分系统

每场对局结束后，玩家根据最终排名获得"胜利积分"（Victory Points）。

积分规则（仅前五名获得）：

| 排名 | 胜利积分 |
|------|----------|
| 第1名 | 10 |
| 第2名 | 7 |
| 第3名 | 5 |
| 第4名 | 3 |
| 第5名 | 1 |
| 第6名及以后 | 0 |

胜利积分跨对局累计，记录在玩家持久化数据中。单机模式受限于没有真实对手，所有 AI 均不参与排行榜排名——单机模式不发放胜利积分。

### 排行榜

全体玩家的排行榜按胜利积分降序排列。积分相同则按胜场数降序，再相同则按玩家标识升序兜底。

排行榜周期：每周一 UTC 00:00 重置。重置时所有玩家的胜利积分归零，上一周期的排名数据存档到历史记录（最初可只保留最近两周）。

排行榜查询由客户端在模式入口界面主动拉取，不通过实时通道推送。服务端需提供 RPC 接口返回排行榜列表。

### 跨场数据流

```
对局结束 → RoomRuntime 计算排名 → 按排名发放胜利积分 → IUserGrain 持久化
                                                              ↓
客户端拉取排行榜 ← ILeaderboardGrain 聚合排序 ← 读取全体用户胜利积分
```

胜利积分存储在 `IUserGrain` 的 `UserState` 中，作为用户持久化状态的一部分。排行榜查询通过独立的 `ILeaderboardGrain`（或 `ILeaderboardService`）聚合全部用户的积分并排序，不直接在客户端遍历用户列表。

## AI 规则

AI 的目标是让局内始终有足够活动目标，而不是追求复杂智能。

AI 行为优先级：

- 小球优先找附近食物。
- 优势足够时追逐更小的玩家。
- 附近有更大威胁时优先远离。
- 被压迫时向相对安全的开阔区域移动。

目标人数当前为 `4`：

- 单机模式创建 1 个本地玩家，再补足 AI。
- 联机模式真人加入房间后，由服务端补足 AI。

AI 不参与胜利积分排行。在排行榜查询和排名统计中过滤所有以 `AI` 为前缀的 bot 账号。

## 共享玩法内核

`samples/Agar.Unity/Shared/Gameplay/ArenaSimulation.cs` 是玩法规则的唯一来源。

它负责：

- 玩家移动。
- 质量影响速度。
- 食物刷新和成长结算。
- 大球吞小球判定。
- AI 决策。
- 死亡、复活和胜负判定。
- 世界状态快照生成。

`Shared` 只能关心协议、配置、规则和状态，不应该依赖 Unity、服务端托管、传输层、数据库、存档或界面。

胜利积分的分值常量（`VictoryPointsByRank`）属于配置层，放在 `ArenaConfig` 或新增的 `ProgressionConfig` 中，不与模拟逻辑耦合。

## 客户端边界

Unity 客户端负责：

- 启动菜单和模式切换。
- 本地单机模拟。
- 联机输入发送。
- 世界状态接收和渲染。
- HUD、玩家表现、食物表现和结算界面。
- 控制连接、实时连接、断线处理和可靠推送确认。
- 本地元进度展示和存储。
- 排行榜拉取和展示。

重要代码位置：

- `Client/Assets/Scenes/Gameplay.unity`：场景来源。
- `Client/Assets/Scripts/Rpc`：传输创建和生成的 RPC 访问入口。
- `Client/Assets/Scripts/Gameplay`：主流程、同步、视图、HUD 和本地进度。

依赖方向：

```txt
Shared -> ULinkRPC.Generated -> SampleClient.Rpc -> SampleClient.Gameplay
```

`Gameplay` 可以依赖 RPC 辅助代码；RPC 辅助代码不能反向依赖玩法界面代码。

已知客户端重构目标：

- `DotArenaGame` 仍是主要应用协调器。
- `DotArenaSceneUiPresenter` 仍偏大，并负责较多运行时控件构建。
- `DotArenaMetaProgression` 已拆分完成，本轮需将本地 mock 排行榜替换为服务端真实数据。

## 服务端边界

`samples/Agar.Unity/Server/Server` 是 RPC 网关和房间运行时宿主。

当前职责：

- 控制面 RPC：登录、登出、匹配和低频业务接口。
- 实时面 RPC：对局输入和实时会话绑定。
- 维护网关本地的在线会话和回调对象。
- 通过 `RoomRuntimeHost` 和 `RoomRuntime` 承载房间运行时。
- 推送世界状态、死亡事件、结算事件和匹配状态。
- 对局结束时按排名发放胜利积分到 `IUserGrain`。

`samples/Agar.Unity/Server/Silo` 承载 Orleans grains。

当前职责：

- 用户身份和胜场持久化。
- 玩家会话状态。
- 匹配队列状态。
- 房间分配和房间快照状态。
- 用户胜利积分持久化（`UserState.VictoryPoints`）。
- 排行榜聚合查询（`ILeaderboardGrain`）。

PostgreSQL 是 Orleans 集群成员表和 grain 状态的持久化后端。本地内存 grain 存储和只靠本机集群的方式不再是当前设计目标。

排行榜 grain 职责：

- 接收排行榜查询请求，聚合全部用户 grain 的胜利积分。
- 按积分降序、胜场降序、玩家标识升序排序后返回 top N。
- 周一 UTC 00:00 触发重置：清空所有用户的 `VictoryPoints`，归档上周数据。
- 排行榜 grain 自身状态只存储当前周期标识（`CurrentWeekId`）和上一周期快照，不存储全量用户数据。

## 联机流程

控制连接流程：

1. 客户端连接控制面 RPC。
2. 客户端登录。
3. 客户端发起匹配。
4. 网关调用匹配 grain。
5. 匹配和房间 grain 分配房间与运行时网关。
6. 网关可靠推送匹配状态，并携带实时连接信息。

实时连接流程：

1. 客户端打开实时 RPC 连接。
2. 客户端用玩家、会话、房间和对局令牌调用 `AttachRealtimeAsync`。
3. 运行时网关登记实时回调。
4. 客户端通过实时 RPC 发送输入。
5. 房间运行时通过实时回调广播世界状态。

排行榜查询流程：

1. 客户端在登录后或模式入口界面通过控制面 RPC 请求排行榜。
2. 网关将请求转发到 `ILeaderboardGrain`。
3. Leaderboard grain 检查当前周期（若已过周一 00:00 则触发重置）。
4. 聚合所有用户胜利积分，排序后返回 top N。
5. 网关将结果返回客户端渲染。

联机同步边界：

- 客户端发送输入。
- 服务端推进模拟。
- 服务端广播快照。
- 客户端对玩家位置做插值，减少快照跳动。

客户端输入消息包含：

```txt
InputMessage
{
    playerId
    moveX
    moveY
    dash
    tick
}
```

服务端广播的世界状态包含：

```txt
WorldState
{
    tick
    respawnDelaySeconds
    players[]
    pickups[]
}
```

`players[]` 包含位置、速度、生死状态、积分、质量、半径和移动速度。`pickups[]` 描述当前仍在地图上的食物。

## 分布式边界

已经分布式或持久化的部分：

- Orleans 成员表和 grain 状态使用 PostgreSQL。
- 匹配队列状态在 Orleans 中。
- 房间分配携带明确的运行时网关信息。
- 客户端收到明确的实时连接目标，不假设控制网关一定拥有房间。
- 实时绑定不再要求本地已有控制连接回调。
- 胜利积分存储在用户 grain 中，跨网关读写均通过 Orleans 客户端。

仍然局限在单个网关进程内的部分：

- 活跃 RPC 回调对象。
- 活跃房间模拟。
- 世界状态广播扇出。
- 部分断线、登出和离房清理语义。

下一步分布式架构重点是网关到网关的输入和事件路由。候选方式包括 Orleans streams、Orleans observers 或 Redis 发布订阅。Redis 只有在明确所有权、顺序和失败模型后才应该进入实时玩法路径。

## 表现原则

- 玩家显示大小跟随 `Radius`。
- HUD 强调名字、质量、排名和存活状态。
- 食物要小且数量多。
- 界面文案使用质量、排名、成长和存活语义，不使用冲刺或技能强化作为核心表达。
- 玩家碰撞和成长可以有果冻感表现，但眩晕、击退不应被当作当前核心玩法。
- 排行榜界面展示当前周期剩余时间、玩家排名、胜利积分和胜场数。

## 暂不实现

- 分裂。
- 吐质量。
- 病毒或尖刺。
- 组队。
- 持久技能树改造。
- 服务端回滚或预测校正重写。
- 房间运行时自动迁移。
- 没有独立设计说明的 Redis 实时路由。
- 超越两周的历史排行榜归档。
